# 雙層架構落地藍圖

> 本文件是 Decision Compass 的主架構藍圖。
> 每一個設計決策都對應到 [`RESEARCH-FOUNDATION.md`](RESEARCH-FOUNDATION.md) 的證據。
> 立場：**嚴格批判但保留可用內核**——只建在有證據的地基上。

---

## 0. 設計第一原則：兩條腿，一道防火牆

研究最重要的啟示是：**統計預測**與**符號反思**是兩種本質不同的東西，
混在一起會互相污染（符號的「神祕感」會讓人高估概率，概率的「數字」會讓符號顯得像預言）。

所以架構的第一原則是**物理隔離**：

```
                    ┌──────────────────────────────────┐
   使用者的決策問題  │         輸入正規化層               │
   （要不要換工作？） │   把模糊問題 → 結構化決策物件      │
                    └────────────┬─────────────────────┘
                                 │
            ┌────────────────────┴────────────────────┐
            ▼                                          ▼
┌───────────────────────────┐          ┌───────────────────────────┐
│   LAYER 1                  │          │   LAYER 2                  │
│   統計概率引擎              │   ╳╳╳    │   結構化反思層             │
│   Probability Engine       │ 防火牆   │   Reflection Protocol      │
│                            │   ╳╳╳    │                            │
│   「會怎樣」的概率與區間    │ 不互灌   │   「你漏想了什麼」的反思    │
└─────────────┬──────────────┘          └─────────────┬─────────────┘
              │                                        │
              └──────────────────┬─────────────────────┘
                                 ▼
                    ┌──────────────────────────────────┐
                    │         整合輸出層                │
                    │  並列呈現：概率事實 + 反思清單     │
                    │  決策仍由「人」做，工具不替你決定   │
                    └────────────┬─────────────────────┘
                                 ▼
                    ┌──────────────────────────────────┐
                    │   決策日誌 + 事後校準（閉環）      │
                    │  記錄當下預測 → 追蹤真實結果       │
                    └──────────────────────────────────┘
```

**防火牆規則（硬性）**：
- Layer 2 的符號輸出**永遠不能**成為 Layer 1 概率計算的輸入。
- Layer 1 的概率數字**不能**被 Layer 2 重新詮釋成「卦象顯靈」。
- 兩層在 UI 上必須**視覺區隔**，並各自標註「這是統計估計」/「這是反思提示，無預測力」。

## 1. Layer 1：統計概率引擎

### 1.1 它做什麼
針對「**有大量可比樣本**」的決策維度，給出概率分佈與風險區間。
例：換工作 → 該行業 N 年留任率分佈、薪資變動分位數、後悔率的參照類別基準。

### 1.2 它**不**做什麼（誠實邊界）
- ❌ 不預測「你個人」的單一結果（個體高後果天花板 R²<0.2，見 RF §2）。
- ❌ 不對長尾／黑天鵝／結構性斷裂事件給點估計。
- ✅ 只給**群體層的概率分佈**，並強制顯示信賴區間與「個體不可化約誤差」警語。

### 1.3 技術構成
| 模組 | 做法 | 證據依據 |
|------|------|----------|
| 參照類別預測 (Reference Class Forecasting) | 把使用者問題映射到歷史基準率資料庫 | Outside view（RF §4） |
| 概率校準層 | 用 Brier score 持續校準，避免過度自信 | LLM 群體預測 Brier 0.20（RF §2） |
| 不確定性量化 | 永遠輸出區間而非點值；標註資料稀疏度 | 個體天花板（RF §2） |
| 黑天鵝守門員 | 偵測到「樣本過稀／非平穩」時拒絕給概率，改走 Layer 2 | 長尾不可學（RF §2） |

### 1.4 為什麼可以信這條腿
人類移動等群體行為可預測性上限達 80–90%（Song 2010, Lu 2013）；
LLM 群體預測已能媲美人類預測者（Schoenegger 2024）。**群體粗粒度概率是有證據的能力。**

## 2. Layer 2：結構化反思層（產品的真正差異化）

### 2.1 核心洞察
研究證明：符號層的**真價值不在易經、不在預測，而在「強迫結構化反思」**——
而這恰好是去偏誤文獻**唯一有 RCT 撐腰**的東西（RF §4）。

所以 Layer 2 的引擎**不是占卜，是一套去偏誤協議**。易經 64 卦只是它的**易讀符號外殼**。

### 2.2 內建的「有 RCT 證據」去偏誤要素（必須全部實作）
| 要素 | 機制 | 證據 |
|------|------|------|
| **Consider-the-Opposite（考慮相反）** | 強迫使用者寫出「這決定為什麼會錯」 | Lord et al. 1984；Morewedge 2015 中–大效果（RF §4） |
| **Premortem（事前驗屍）** | 假設一年後此決定徹底失敗，回推原因 | Klein/Kahneman（RF §4） |
| **Outside View（外部視角）** | 把「我這次」換成「像我這樣的人通常」 | Kahneman-Lovallo；接 Layer 1 基準率 |
| **決策日誌 + 事後校準** | 記錄當下信念，事後對照，對抗確認偏誤 | 持續性結構設計（RF §4 限制的補救） |

### 2.3 易經符號層的**正確**用法（與唯一合法用法）
- ✅ 64 卦 = 一套**人類易讀的 64 格降維符號標籤**，用來**觸發**上述反思（例：抽到「困卦」→ 觸發 premortem「什麼會讓你被困」）。
- ✅ 卦象提供**敘事框架與隱喻**，降低「面對不確定的決策癱瘓」。
- ❌ **嚴禁**：宣稱卦象能預測結果、宣稱卦象有數學神祕計算力。
  - 研究實測：王文卦序拿去調制神經網路訓練→**毫無計算優勢、甚至變差**（RF §3）。
  - 「64 卦對應遺傳密碼」等=數字命理附會，已被驗證駁回（RF §3）。

### 2.4 去巴納姆防護（Anti-Barnum Guardrails）— 不可妥協
這是讓本工具**不淪為占卜騙局**的關鍵。每一條都直接針對 Forer 效應（RF §3）：

1. **禁止通用模糊語句**：所有反思提示必須**引用使用者自己輸入的具體事實**，不得出現「你有時外向有時內向」這類巴納姆語句。
2. **強制證偽**：每個提示都要求使用者**主動寫出反例**，而非被動點頭認同（confirmation bias 的解藥）。
3. **明示無預測力**：UI 每次都標註「以下是反思提示，不是預測」。
4. **可證偽的決策日誌**：記錄具體、可被現實打臉的預測，事後回看——這讓「準確感」無處藏身。

## 3. 整合輸出層

```
┌───────────────────────────────────────────────────────────┐
│  決策：要不要接受這個外派機會？                            │
├───────────────────────────────────────────────────────────┤
│  📊 Layer 1 統計事實（群體基準，非你個人預測）            │
│     • 類似外派 3 年留任率：62% [信賴區間 54–70%]          │
│     • 後悔率參照基準：約 30%                               │
│     • ⚠️ 個體結果不可化約誤差大，以下非命運預測            │
├───────────────────────────────────────────────────────────┤
│  🧭 Layer 2 反思提示（無預測力，幫你想全）                │
│     • 考慮相反：寫下 3 個「接受後會後悔」的具體情境        │
│     • 事前驗屍：假設兩年後此決定失敗，最可能的原因是？     │
│     • 外部視角：和你處境相似的人，通常忽略了什麼？         │
├───────────────────────────────────────────────────────────┤
│  ✍️ 你的決策日誌（事後校準用）                            │
│     我的決定：___  我預測 1 年後的滿意度：__/10            │
└───────────────────────────────────────────────────────────┘
        決策由你做。工具給概率與盲點，不給答案。
```

## 4. 技術堆疊建議（初版）

| 層 | 選型方向 | 備註 |
|----|---------|------|
| 概率引擎 | 參照類別資料庫 + 校準後的統計/ML 模型 | 起步可用既有公開統計資料 + 簡單貝氏更新 |
| 反思協議 | LLM 驅動，但**嚴格 prompt 約束**在去偏誤協議內 | LLM 只當「反思引導者」，禁止當「預言家」 |
| 符號介面 | 64 卦 → 反思情境的靜態映射表 | 純查表，無任何「演算」 |
| 決策日誌 | 結構化儲存 + 到期提醒回訪 | 閉環校準的命脈 |
| 防火牆 | 兩層獨立服務 / 模組，介面層才合流 | 架構上保證不互灌 |

> 注意 neuro-symbolic 混合架構（RF §3）有真實但**狹窄、非全面**的優勢——
> 初版**不需要**上 neuro-symbolic；先把「統計概率 + 去偏誤協議」這兩條有證據的腿做扎實。

## 5. 三個必須避開的陷阱（研究明確警告）

1. **把符號當預測引擎** → 必落入巴納姆效應 + 確認偏誤（RF §3）。防火牆 + 去巴納姆防護就是為此而設。
2. **把群體概率當個體命運** → 個體天花板 R²<0.2（RF §2）。永遠標註區間與不可化約誤差。
3. **以為去偏誤一勞永逸** → 去偏誤遷移弱、真實生活效果未證實（RF §4）。所以要靠**持續性的決策日誌**，而非單次介入。
